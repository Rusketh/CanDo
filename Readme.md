# CanDo

**CanDo** is a C-style scripting language built around a single design
goal: be small enough to embed anywhere, but built on a real
multi-threaded runtime from day one.

It is a complete pipeline — lexer, Pratt parser, bytecode compiler, and a
stack-based virtual machine — implemented in C11 and exposed as both a
standalone interpreter (`cando`) and a single-header embedding library
(`libcando`).  Source files use the `.cdo` extension and compile to
bytecode for an instruction-dispatch loop driven by GCC computed gotos.

---

### Why CanDo exists

CanDo was created to replicate the success of **Lua** — a small,
embeddable, batteries-fits-in-a-shoebox scripting language that quietly
ended up inside countless games, tools, and platform frameworks — but
with one structural change: a **real, first-class threading environment
from day one**, instead of a single-threaded core that the embedder is
expected to wrap a lock around.

The runtime was designed from the start around real OS threads and a
value system that survives concurrent access — not bolted on later, not
fenced behind a global interpreter lock.

That means:

- Every value type the script sees is safe to share between threads.
- Heap objects are referred to by relocatable handles, so the GC can
  move them without invalidating live values on another thread's stack.
- The stdlib (sockets, files, HTTP, channels, streams, mutexes via
  `object.lock`) is written to the same threading contract as user
  scripts, not against a single-threaded fast path with locks tacked
  on.

The result is a small, embeddable scripting language whose concurrency
story is the same in C and in `.cdo` — one runtime, real threads, no
global interpreter lock.

The architecture, language design, and module specifications were

## Quick start

### Build

CMake is the supported build system; a GNU `Makefile` is provided as a
fallback.  OpenSSL and pthreads are required.

```bash
cmake -B build && cmake --build build
./build/cando tests/scripts/hello.cdo
```

Or with `make`:

```bash
make            # builds tests + the cando interpreter
make cando      # interpreter only
make test       # runs the full test suite
```

### Hello world

```cdo
// hello.cdo
print("Hello, CanDo!");
```

```bash
./cando hello.cdo
```

### Spawn a thread

```cdo
VAR worker = thread {
    thread.sleep(50);
    RETURN "done";
};

print(await worker);            // "done"
```

### Embed in C

```c
#include <cando.h>

int main(void) {
    CandoVM *vm = cando_open();
    cando_openlibs(vm);

    if (cando_dofile(vm, "game/main.cdo") != CANDO_OK)
        fprintf(stderr, "%s\n", cando_errmsg(vm));

    cando_close(vm);
    return 0;
}
```

```
gcc myapp.c -lcando -Iinclude -o myapp
```

A full embedding walk-through lives in
[docs/api/embedding.md](docs/api/embedding.md).

## Language at a glance

```cdo
// Variables — VAR is mutable, CONST is write-protected.
VAR  name  = "world";
CONST PI   = 3.14159;

// Strings — three quote styles.
VAR plain    = "double-quoted, escapes like \n work";
VAR raw      = 'single-quoted, multi-line
                no escapes';
VAR template = `hello, ${name}!`;     // backtick interpolation

// Numbers are IEEE-754 doubles.  Booleans are TRUE / FALSE.

// Arrays are 0-indexed and dense.
VAR xs = [1, 2, 3, 4, 5];

// Objects keep insertion order.
VAR person = { name: "Alice", age: 30 };

// Pipes — array transformations.
VAR doubled = xs ~> pipe * 2;          // [2, 4, 6, 8, 10]
VAR evens   = xs ~&> pipe % 2 == 0;    // [2, 4]

// Ranges.
FOR i IN 1 -> 5  { print(i); }         // 1 2 3 4 5
FOR i IN 5 <- 1  { print(i); }         // 5 4 3 2 1

// Functions, closures, multi-return.
FUNCTION minmax(a, b) {
    IF a < b { RETURN a, b; }
    RETURN b, a;
}
VAR lo, hi = minmax(7, 3);

// Classes with prototype-based inheritance and metamethods.
CLASS Vector = (self, x, y) { self.x = x; self.y = y; }
Vector.__add = FUNCTION(a, b) { RETURN Vector(a.x + b.x, a.y + b.y); };

// Threads are a built-in expression form.
VAR results = [];
FOR i IN 1 -> 4 {
    results:push(thread { RETURN i * i; });
}
FOR t OF results { print(await t); }   // 1 4 9 16

// Errors — TRY / CATCH / FINALY (one L) / THROW.
TRY {
    THROW "validation", 422, "missing 'name'";
} CATCH (kind, code, detail) {
    print(kind, code, detail);
}
```

## What's in the box

**Standard library** (always present in the `cando` CLI):

- Core data: `array`, `string`, `math`, `object`
- Serialization: `json`, `yaml`, `csv`
- I/O: `file`, `stream`, `os`, `process`
- Crypto and time: `crypto`, `datetime`
- Networking: `net`, `socket`, `secure_socket`, `http`, `https`
- Concurrency: `thread`
- Runtime control: `gc`, `jit`, `app`, `eval`, `include`, `_meta`

**Binary modules** (loaded with `include()`):

- `sqlite` — SQLite bindings with prepared statements and transactions.
- `sql` — MySQL / MariaDB / PostgreSQL.
- `ldap` — LDAP / Active Directory client.
- `smtp` — SMTP, IMAP, POP3, DKIM, SPF.
- `window` — Cross-platform OS window with an OpenGL context.
- `draw` — 2D drawing primitives layered on `window`.
- `forms` — Native widget tree (Win32 today; stub elsewhere).

Each module is documented next to its source in
[`modules/`](modules/README.md).  Per-function reference for each stdlib
namespace lives in [docs/libraries/](docs/libraries/README.md).

## Repository layout

```
include/cando.h          single public header for embedders
source/
  core/                  values, memory, handles, lock primitives
  object/                heap objects: CdoObject, array, function, thread
  parser/                lexer + Pratt parser → bytecode
  vm/                    bytecode, dispatch loop, bridge layer, disassembler
  jit/                   tracing JIT (counters, IR, native codegen)
  lib/                   standard library
  natives.c              core natives: print, type, toString, inspect
  cando_lib.c            public embedding API (cando_open, cando_dofile, …)
  main.c                 cando CLI entry point
modules/                 binary extension modules (one subdirectory each)
editors/vscode-cando/    VS Code extension (syntax + LSP)
docs/                    documentation (you are here)
tests/
  scripts/               .cdo scripts run by the integration suite
  integration/           shell runner that executes every script
  test_*.c               unit tests for core, object, lexer, parser, vm
```

## Status and version

CanDo currently tracks `1.0.0` — `CANDO_VERSION` in
[`include/cando.h`](include/cando.h).  The numeric form
`CANDO_VERSION_NUM` is `10000` (major × 10000 + minor × 100 + patch).
